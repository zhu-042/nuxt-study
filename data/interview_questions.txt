一、Vue 3 / Nuxt 3 核心
========================================================================

【第1题】Vue 3 的响应式原理是什么？和 Vue 2 有什么区别？

答案：
Vue 3 使用 Proxy 代替 Vue 2 的 Object.defineProperty。
- Proxy 可拦截属性新增/删除/数组下标修改，defineProperty 不行。
- Vue 3 惰性代理，访问到嵌套对象时才代理；Vue 2 递归遍历全部属性。
- ref（基本类型）+ reactive（对象），依赖收集用 WeakMap→Map→Set。
- effect → track（收集）→ trigger（触发更新）。

【第2题】Composition API 相比 Options API 有哪些优势？

答案：
1. 逻辑复用：composables 替代 mixins，无命名冲突。
2. 代码组织：相关逻辑集中，不按 data/methods/computed 分散。
3. TypeScript 友好：返回值可自然推导类型。
4. Tree-shaking：按需引入 API，减小包体积。

【第3题】Nuxt 3 的渲染模式有哪些？

答案：
1. SSR：默认，服务端渲染+客户端水合，适合 SEO 页面。
2. CSR：ssr:false，纯客户端渲染，适合后台管理。
3. SSG：nuxt generate 预渲染静态页面。
4. ISR/SWR：routeRules 配置缓存时间，增量更新。
5. Hybrid：不同路由可配不同渲染策略。

Nuxt 3 常见的渲染模式有 SSR、CSR 和 SSG。SSR 是服务端先渲染出首屏 HTML，
客户端再进行水合，适合 SEO 页面；CSR 是页面主要由客户端 JS 渲染，
适合后台管理系统；SSG 是在构建阶段提前生成静态页面，访问时直接返回静态文件，
适合博客、文档这类内容相对稳定的页面。

【第4题】useFetch 和 useAsyncData 的区别？

答案：
- useFetch = useAsyncData + $fetch 的语法糖，自带请求功能。
- useAsyncData 更通用，接受任意异步函数。
- 共同特性：SSR 数据通过 payload 传递避免重复请求、key 去重、lazy/transform/watch 选项。
- 只能在 setup/生命周期/插件/中间件中使用。

useFetch 可以理解成 useAsyncData 配合 $fetch 的语法糖，
主要用于接口请求，写法更简洁；useAsyncData 更通用，它接收任意异步函数，
所以适合更复杂的数据获取或组合逻辑。两者都支持 SSR 阶段取数并通过 payload 传给客户端，
避免重复请求，也都支持 key 去重、lazy、transform、watch 等选项。

------------------------------------------------------------------------

【第5题】Vue 3 的 Teleport 组件有什么作用？项目中如何使用？

答案：
Teleport 将组件 DOM 渲染到指定节点，不改变逻辑父子关系。
项目中知识库模块：大屏时 Teleport 到分栏右侧面板，小屏时动态切换 to 属性传送到抽屉容器，实现不同布局下组件无缝迁移，避免重新创建和状态丢失。

Teleport 可以根据条件动态把组件挂载到不同的 DOM 容器中，
比如大屏挂到右侧栏，小屏挂到抽屉容器里。这样组件逻辑不变，
只是渲染位置变了，可以避免重复创建组件和状态丢失。

------------------------------------------------------------------------

【第6题】watch 和 watchEffect 的区别？

答案：
- watchEffect：立即执行，自动追踪依赖，无新旧值。
- watch：需指定监听源，可获取新旧值，默认懒执行，支持 deep。 immediate: true 默认执行
- 需要新旧值对比用 watch，自动追踪多个依赖用 watchEffect。

------------------------------------------------------------------------

【第7题】Nuxt 3 中间件的种类和执行顺序？

答案：
1. 全局中间件（.global.ts）：每次路由跳转都执行。
2. 命名中间件：页面 definePageMeta 引用。
3. 行内中间件：definePageMeta 中直接定义。
执行顺序：全局（按文件名排序）→ 页面级。
可返回 navigateTo() 重定向或 abortNavigation() 取消。

【第9题】App Router 和 Pages Router 的区别？

【第10题】React Server Components 和客户端组件的区别？

答案：
Server Components：服务端渲染，不发送 JS 到客户端，可访问数据库/文件系统，不能用 hooks/事件。
Client Components（'use client'）：支持交互，JS 发送到客户端，首次也会 SSR 预渲染。
最佳实践：'use client' 边界尽量下推，只在需要交互处标记。

React Server Components 和 Client Components，
本质上就是服务端组件和客户端组件的区分。前者主要运行在服务端，
适合数据获取和页面渲染，不能处理交互；后者运行在客户端，
支持 hooks 和事件处理，适合有交互的部分

【第11题】ISR（增量静态再生）如何工作？

答案：
1. 构建时预渲染（像 SSG）。
2. 设置 revalidate 时间，缓存过期后先返回旧内容，后台异步重新生成。
3. App Router 中：fetch({ next: { revalidate: 3600 } }) 或 export const revalidate。
项目应用：EssayPass 营销首页 ISR 1 小时缓存，兼顾 SEO 和内容更新。

【第12题】next-intl 国际化方案及你做了哪些优化？

答案：
- [locale] 动态路由段区分语言，middleware 检测语言并重定向。
- 优化：默认语言不带 URL 前缀；307 改 301 永久重定向优化 SEO 权重传递。
- 58 个翻译命名空间按页面/组件粒度拆分减少加载体积。

next-intl 做国际化，通过 [locale] 动态路由段区分不同语言，
并在 middleware 中检测用户语言后做重定向。优化上，一是默认语言不带 URL 前缀，让地址更简洁；
二是把部分语言跳转从 307 调整为 301，更利于 SEO 权重传递；
三是把翻译资源按页面和组件拆成多个 namespace，减少单次加载体积

【第13题】interface 和 type 的区别？

答案：
- interface 支持声明合并，用 extends 扩展；type 不支持合并，用 & 交叉。
- type 更强大：支持联合类型、元组、条件类型、映射类型、基本类型别名。
- 定义对象结构优先 interface，需要联合/工具类型时用 type。

interface 和 type 都能定义类型，定义普通对象时都可以用。
区别是 interface 更适合描述对象结构，支持声明合并，扩展时一般用 extends；
type 更灵活，不仅能定义对象，还能定义联合类型、元组、基本类型别名、交叉类型以及条件类型等复杂类型。
实际开发里，定义对象结构通常优先用 interface，需要更复杂的类型表达时用 type

【第16题】SSR 水合（Hydration）过程及常见问题？

答案：
过程：服务端生成 HTML → 客户端加载 JS → 框架接管 DOM 绑定事件和状态。
常见问题：Hydration Mismatch（服务端/客户端渲染不一致），原因包括 Date.now()、window/localStorage、条件渲染依赖客户端状态。
解决：<ClientOnly> 包裹或 onMounted 中设置客户端状态。

SSR 的 hydration 过程是：服务端先生成 HTML，客户端加载 JS 后由框架接管现有 DOM，并绑定事件和状态，让页面变成可交互。
常见问题是 Hydration Mismatch，也就是服务端和客户端首次渲染结果不一致，
常见原因包括 Date.now()、Math.random()、window、localStorage，以及依赖客户端状态的条件渲染。
解决方式通常是保证首屏渲染一致，或者把纯客户端逻辑放到 <ClientOnly> 或 onMounted 中处理。

帮助 Google 更好理解页面内容，并有机会展示富摘要，从而提升搜索结果点击率。

【第19题】SSR 内外网地址分离的作用？

答案：
服务端渲染用内网地址访问 API，客户端用公网地址。
好处：内网延迟极低（<1ms），降低 TTFB；节省带宽；增强安全性。
实现：根据 process.server 在请求拦截器中自动切换 baseURL。

------------------------------------------------------------------------

【第20题】301 和 307 重定向的区别？为什么国际化要用 301？

答案：
301 永久重定向：搜索引擎转移权重到新 URL，浏览器缓存。
307 临时重定向：搜索引擎保留原 URL 权重，不缓存。
next-intl 默认 307（/about → /en/about），搜索引擎不传递权重。改 301 后权重正确传递，提升收录。

【第21题】SSE 和 WebSocket 的区别？

答案：
SSE：单向（服务器→客户端），基于 HTTP，自动重连，只支持文本。适合 AI 流式输出、通知推送。
WebSocket：双向，独立协议，需自行重连和心跳，支持二进制。适合聊天、语音、实时协作。
项目：AI 对话用 SSE，语音识别用 WebSocket。

流式文本 → Markdown 解析 → 代码高亮 / KaTeX / 自定义引用。
注意：组件卸载时 AbortController 取消请求；节流控制渲染频率；智能滚动策略。

【第23题】Web Audio API 实时语音识别流程？

答案：
getUserMedia 获取麦克风 → AudioContext + ScriptProcessorNode 处理音频 → 编码降采样到 16kHz → WebSocket 流式发送至 ASR 服务 → 接收增量识别结果 → 更新文本。

六、文件上传与存储
========================================================================

【第24题】SHA256 文件秒传原理？

答案：
前端用 Web Crypto API（crypto.subtle.digest）计算文件 SHA256 → 发送哈希到服务端比对 → 已存在则直接复用（秒传），不存在则正常上传。
大文件可用 Web Worker 避免阻塞主线程。

------------------------------------------------------------------------

【第25题】六步文件上传流程的设计？

答案：
1. SHA256 计算 → 2. 秒传检查 → 3. 获取 COS 预签名 URL → 4. 对象存储直传（带进度）→ 5. 后端注册元信息 → 6. 关联知识库。
步骤 1-2 避免重复上传；步骤 3-4 文件直传 COS 减轻服务端压力；步骤 5-6 解耦存储与业务。
额外：WPS 格式自动转换、空间容量校验、上传后轮询处理状态。

------------------------------------------------------------------------

【第26题】预签名 URL 直传的优势？

答案：
后端用密钥生成临时签名 URL → 前端直传对象存储 → 不经过业务服务器。
优势：降低服务端带宽和负载、密钥不暴露给前端、URL 有过期时间、直连 CDN 速度快、支持上传进度监控。

【第32题】游标分页 vs offset 分页？

答案：

游标：WHERE id > cursor LIMIT N，性能稳定，无数据偏移，天然支持无限滚动。
项目：资讯列表用游标分页，因内容频繁更新，offset 会导致重复文章。

【第35题】智能滚动控制策略如何实现？

答案：
AI 流式输出时需要自动滚动，但用户手动上翻时不能打断阅读。
实现：
1. 节流监听 scroll 事件，判断用户是否在底部（距底部 < 阈值）。
2. 维护 isAutoScroll 标志：在底部时为 true。
3. SSE 收到新 token 时，isAutoScroll 为 true 才自动 scrollTo。
4. 用户手动上翻，isAutoScroll 变 false，暂停自动滚动。
5. 用户滚回底部时恢复自动滚动。

【第40题】归因追踪系统如何实现？

1. 首次访问采集 UTM 参数（utm_source/medium/campaign）、广告点击 ID（gclid/fbclid）、来源页（document.referrer）、落地路径。
2. 写入 localStorage，设置 30 天有效期。
3. 用户注册时自动携带归因数据发送给后端。
4. 后端据此分析各渠道的获客效果和 ROI。

【第41题】CSS Container Queries 是什么？和 Media Queries 的区别？

答案：
Media Queries：基于视口（viewport）尺寸响应。
Container Queries：基于父容器尺寸响应，组件级响应式。
项目应用：知识库分栏布局中，组件根据所在容器宽度自适应，而非视口宽度，分栏拖拽时组件自动调整布局。

【第42题】EventBus 在 Vue 3 中如何实现？有什么替代方案？

答案：
Vue 3 移除了 $on/$emit 实例方法，不再内置 EventBus。
替代方案：
1. 使用 mitt 或 tiny-emitter 等第三方库。
2. 使用 provide/inject 传递事件。
3. 使用 Pinia store 共享状态。
项目中用于翻译与 PDF 联动等跨组件通信场景。

【第43题】Sentry 前端监控的接入和使用？

答案：
1. 安装 @sentry/nextjs，配置 DSN 和环境信息。
2. 自动捕获未处理异常和 Promise rejection。
3. 自定义错误上报：Sentry.captureException(error)。
4. 性能监控：自动采集页面加载、API 请求耗时。
5. Source Map 上传：构建时上传 sourcemap，线上报错可定位到源码行。
6. 环境区分：dev/staging/production 不同上报策略。

【第44题】前端如何做错误处理？

答案：
1. 全局错误边界：Vue 的 app.config.errorHandler / React 的 ErrorBoundary。
2. API 请求：Axios/fetch 拦截器统一处理 HTTP 错误码。
3. 异步错误：try-catch + async/await。
4. Nuxt：error.vue 页面处理全局错误、createError 抛出错误。
5. Next.js：error.tsx 处理路由级错误、not-found.tsx 处理 404。
6. Sentry 上报：关键流程主动上报错误信息和上下文。

【第45题】如何实现组件的异步懒加载？

答案：
Vue 3：defineAsyncComponent(() => import('./Comp.vue'))
Nuxt 3：组件名加 Lazy 前缀（<LazyComp />）自动懒加载。
React：React.lazy(() => import('./Comp')) + <Suspense fallback>。
Next.js：next/dynamic 支持 ssr: false 选项。
项目中知识库 Tab 系统采用异步组件 + 缓存策略，切换不销毁已加载组件。

【第46题】Tiptap 富文本编辑器的使用经验？

Tiptap 基于 ProseMirror，headless 无预设 UI，高度可定制。

【第47题】项目中如何处理多端适配？

万能小in 支持 Web 和小程序 WebView。

2. 交互适配：鼠标/触摸双套事件处理（如浮动帮助系统的拖拽）。

【第48题】如何保证 SSR/CSR 状态一致性？

答案：
1. 避免在渲染中使用浏览器 API（window/localStorage）。
2. 需要客户端状态时，用 onMounted / useEffect 延迟设置。
3. Pinia 状态通过 payload 自动序列化/反序列化。
4. 关键状态同步到 Cookie（如团队标识），SSR 时从 Cookie 读取。
5. 使用 <ClientOnly> / 'use client' 标记仅客户端渲染的内容。

【第49题】如何做代码分割和按需加载？

答案：
1. 路由级分割：Nuxt/Next.js 自动按路由分割。
2. 组件级：动态 import + defineAsyncComponent / React.lazy。
3. 第三方库：需要时才 import（如导出 Excel 时才加载 xlsx）。
4. UI 组件库按需引入（Ant Design Vue 的 unplugin-vue-components）。
5. 图片/视频懒加载。
